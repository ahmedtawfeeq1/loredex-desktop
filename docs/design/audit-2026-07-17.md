## Q3+Q4 — Findings (severity, evidence, fix)

**P1 — Clients nav inbox badge lost in v3 rewrite.** `App.tsx:92-96` computes `clientsPending = inboxPending(fleet)` (the documented badge) but never renders it — `nav` and `clientsPending` are dead since nav moved into SideNav, which badges only handoffs and agents (`SideNav.tsx:202-208`, collapsed rail `:160-162`). Fleet-wide pending-consumption count is invisible from the nav — the core fleet-monitoring signal. Fix: in SideNav render a warn-tinted `nav-count-pill` on the clients row from `inboxPending(useDex(s => s.fleet))` (both expanded and collapsed rails); delete the dead code in App.tsx.

**P2 — Clients row mislabeled ⌘8 (duplicate of Settings).** Expanded SideNav filters Settings out then numbers by the filtered index (`SideNav.tsx:188,194-195,207`): on agent-ops Clients lands at i=7 → title "Clients (⌘8)", `aria-keyshortcuts Meta+8`, nav-num "8" — while the pinned Settings row hardcodes ⌘8 (`SideNav.tsx:223-229`) and the real binding is ⌘9 (`registry.ts:107-110`, verified by `registry.test.ts:67-80`). Two rows say "8"; pressing the advertised key opens Settings. Fix: number rows by their index in the unfiltered `visibleViews()` list, not the filtered one; also make the footer `keys 1–8` (`SideNav.tsx:231`) read `1–9` on agent-ops, and give the collapsed rail's clients row its ⌘9 hint (`SideNav.tsx:155` caps hints at `i < 8`).

**P2 — SideNav shelves never got the agent-ops relabel VaultTree got.** `shelvesFrom` works off the same grouped tree so client dexes do render shelves (flat fallback at `SideNav.tsx:54-55`), but the header fallback reads "PROJECTS" (`SideNav.tsx:77` — `(product ?? 'projects')`) and every shelf carries a hardcoded `product` tag (`SideNav.tsx:78`) even when the group is a Manager. VaultTree handles this (`VaultTree.tsx:139-145`); the shelf missed parity. Fix: read `useDex(s => s.type)` in `Shelf`; fallback label `clients`, tag `manager` on agent-ops.

**P2 — Shelf rows show research facts and drill to the wrong place on agent-ops.** Row badges are open-handoff counts and stale-brief dots (`SideNav.tsx:72,83-85,104-108`) — research semantics; a client's `inboxCount` never surfaces here. Click routes to Atlas `drillProject` (`SideNav.tsx:93-95`), but the Atlas graph is markdown-only (`src/core/atlas.ts:1078` `listMarkdownFiles`, `:198` `\.md$` matcher), so drilling a data-file-heavy client shows a sparse/near-empty graph instead of its fleet page. Fix: on agent-ops, badge rows with `client.inboxCount` (amber) and route click to `setView('clients')` + `selectClient(slug)`.

**P2 — Agent-ops inbox has no consume affordance (and no item list).** ClientPage shows only a count + "Consume by moving each file to its proper home" (`ClientPage.tsx:255-261`) — a hand-edit instruction with no action, and `ClientInfo` carries only `inboxCount`/`inboxOldestMs`, no item paths (`node_modules/loredex/dist/lib.d.ts:149-151`), so items can't even be listed or opened. Fix: extend lib `scanFleet` to include inbox item relPaths, render them as reader-open rows, and wire a consume/route action (CLI-backed, e.g. `loredex route`) instead of prose.

**P2 — Today/dashboard is fleet-blind on agent-ops.** TodayView/insights/dashboard-data have zero agent-ops awareness (grep for `agent-ops|dexType|manager` across `TodayView.tsx`, `dashboard-data.ts`, `AtlasView.tsx`, `PlanView.tsx`, `AgentsView.tsx` = 0 hits); the "day's work surface" shows handoff velocity and project briefs but never pending inboxes, schema errors, or workspace drift. Fix: on agent-ops add a fleet attention rail card (pending count, error-lint count, oldest inbox age) built from the already-loaded `useDex` store — no new IPC.

**P3 — `manager:` facet filters but is undiscoverable.** The query parser accepts it (`query-parser.ts:27`) and filtering works (`facets.ts:67-68`), but `aggregateFacetValues` emits only projects/topics/types/statuses (`facets.ts:96-127`; `types.ts:103`), so the search UI never offers manager values. Fix: add `managers[]` to `FacetValues` from the products manifest; show its chips only on agent-ops.

**P3 — Lint errors have no global surfacing.** Lints are fetched fleet-wide (`dex.ts:39-49`) but appear only per client card/page (`ClientsView.tsx:46-48,79`; `ClientPage.tsx:216`). Fix: aggregate error count into a rust pill on the Clients view header (or nav row) so schema drift is visible without opening each client.

**P3 — Agent-ops surfaces sit outside the design-fidelity guard.** `design-fidelity.test.ts` has no rules for clients/shelf surfaces (grep = 0), and ClientsView/ClientPage inline CSS carries stale pre-v3 hex fallbacks (`var(--rust, #a33f2e)` `ClientsView.tsx:41`, `ClientPage.tsx:27,45,47`; `var(--ok, #2e6e5e)` `ClientPage.tsx:56`) that would silently mask token drift — tokens do exist today (`assets/loredex-v3.css:36-38`) so rendering is correct. Fix: extend fidelity rules to the clients CSS blocks and drop the raw-hex fallbacks.

**Research type:** no lost surface or material affordance gap found — the v3 eight (Today/Inbox/Plan/Reader/Atlas/Agents/Activity/Settings) plus absorbed Search/Contracts all serve knowledge flow, and research dexes pay zero agent-ops cost (type loads once, no fleet IPC — `dex.ts:29-37`).
